## 执行顺序

**两阶段自检**：

| 阶段 | 范围 | 耗时参考 | 通过条件 |
|------|------|----------|----------|
| 第一阶段（脚本可查） | QC-02/05/08/09/11/12/17/18/20/21/23/24/25/27 | ~2 min | 14 项全部通过 |
| 第二阶段（LLM 可查） | QC-01/03/04/06/07/10/13/14/15/16/19/22/26 | ~10 min | 13 项全部通过 |

第一阶段任一失败 → 修复后重新执行第一阶段，通过后再进入第二阶段。

## 自动化检查指引

可脚本检查的 QC 项（标注"脚本可查"）用以下方式验证：

| QC | 检查方法 |
|----|----------|
| QC-02 | 提取 §7.1 偏移地址列，检查重复值 |
| QC-05 | 统计 §14 REQ 行数 vs 上游 REQ 总数，计算覆盖率 |
| QC-08 | `ls /ds/doc/fs/` + `ls /ds/doc/fs/tmp/` 验证文件存在 |
| QC-09 | `d2 --layout <file.d2>` 编译验证 + Wavedrom JSON 格式校验 |
| QC-11 | 提取 §8.4 合计行与 §8.1~8.3 数值比较 |
| QC-12 | 提取 §7.2 每个寄存器 bit 范围，检查 0~(MSB) 全覆盖 |
| QC-17 | 提取 §5.x 编号序列，检查连续性 |
| QC-18 | `find /ds/doc/fs/tmp/ -name "*.d2" -exec d2 --layout dagre {} \;` |
| QC-20 | grep §5.x.2 中 `\[\d+:\d+\]` 硬编码模式 |
| QC-21 | 提取 §3.1 F-xxx 与 §4.1 REQ-xxx，交叉比对 |
| QC-23 | 提取文档中所有 `![...](xxx.png)` 引用，检查每个 PNG 文件是否存在于文档同级目录或 `/tmp/` 子目录。伪代码：`for ref in grep('!\[.*\]\(.*\.png\)', doc): assert exists(ref.path)` |
| QC-24 | **D2 模块名一致性**：提取 D2 脚本中所有 `label` 字段，检查文档 §3.1 或 §5.x 中是否有对应描述。伪代码：`d2_labels = extract_labels(d2_file); for label in d2_labels: assert label in doc_section_3_1 or label in doc_section_5` |
| QC-25 | **D2 信号名一致性**：提取 D2 脚本中所有连线标签，检查文档 §5.x.2 端口列表或 §6.2 信号列表中是否有同名信号。伪代码：`d2_signals = extract_edge_labels(d2_file); for sig in d2_signals: assert sig in doc_ports` |
| QC-26 | **LLM 图文描述一致性审查**：LLM 读取 D2 脚本和对应文档章节，对比以下维度的一致性：① D2 脚本中的子模块连接关系 vs 文档 §3.1 关键特性表或 §5.x 子模块描述；② D2 脚本中的数据流向 vs 文档 §4.3 数据流描述。输出格式：每个维度给出 ✅一致/❌不一致 判断，不一致项列出具体差异。**注意**：LLM 不检查 PNG 图片本身，仅检查生成脚本与文字描述的一致性。 |
