# 变更传播规则

> 当上游文档或规范发生变更时，chip-code-writer 需按以下规则判断是否触发级联更新。

---

## 触发条件

| 变更源 | 变更类型 | 影响范围 | 触发行为 |
|--------|----------|----------|----------|
| 微架构文档 §4.1 | 端口增删/位宽变更 | RTL 端口 + SDC + SVA + TB | **强制重新实现** |
| 微架构文档 §5.3 | FSM 状态增删/转移变更 | RTL 控制逻辑 + SVA | **强制重新实现** |
| 微架构文档 §5.5 | FIFO 深度/位宽变更 | RTL FIFO 模块 | **强制重新实现** |
| 微架构文档 §5.6 | CBB 参数变更 | RTL CBB 实例化 | **强制重新实现** |
| 微架构文档 §6 | 时钟频率变更 | SDC 约束 | **强制更新 SDC** |
| 微架构文档 §10 | 验证场景变更 | SVA 断言 | **强制更新 SVA** |
| 微架构文档 §3/§8/§11~13 | 描述性内容变更 | 无 | **无需更新** |
| 编码规范 | 新增/修改规则 | RTL 代码风格 | **自检阶段重新执行** |
| CBB 文档 | 版本升级 | RTL CBB 实例化 | **检查兼容性，按需更新** |

---

## 传播流程

```
变更检测 → 影响分析 → 范围确认 → 级联更新 → 质量门禁重跑
```

### Step 1: 变更检测
- **自动检测**：运行 `bash .claude/shared/change-detect.sh <microarch_doc_path> [base_commit]`
- 脚本自动对比 git diff，识别变更章节并输出 JSON 影响报告
- **手动检测**：用户告知变更内容，人工对比差异

### Step 2: 影响分析
- 按上表判断变更影响范围
- 输出影响文件列表

### Step 3: 范围确认
- Critical 变更（端口/FSM/FIFO/CBB）→ 暂停，列出影响范围，等用户确认
- Minor 变更（描述性内容）→ 直接跳过

### Step 4: 级联更新
- 按影响范围重新执行对应 Skill
- 仅更新受影响的文件，不重写全部

### Step 5: 质量门禁重跑
- Lint + 综合 + 自检 重新执行
- 确认变更后代码仍通过所有门禁

## 版本号更新规则

| 变更类型 | 版本段 | 示例 |
|----------|--------|------|
| 端口/FSM/FIFO 深度变更 | major +1 | v1.0.0 → v2.0.0 |
| 新增功能/修改逻辑 | minor +1 | v1.0.0 → v1.1.0 |
| Bug 修复/优化 | patch +1 | v1.0.0 → v1.0.1 |
