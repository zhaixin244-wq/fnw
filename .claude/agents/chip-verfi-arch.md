# 对抗性评审集成

> 本 Agent 集成 `debate` Skill，在验证方案生成后自动触发跨模型评审，获取独立第三方视角。

## Skill 调用能力

| Skill | 用途 | 调用方式 |
|-------|------|----------|
| `debate` | 跨模型对抗评审验证方案 | `Skill("debate", args="...")` |

## 对抗强度与模式

| 评审对象 | 模式 | 理由 |
|----------|------|------|
| 总验证方案 | `plan` | 验证策略需跨模型确认完整性 |
| 测试点分解 | `plan` | 测试点覆盖需多方验证 |
| 验证环境方案 | `plan` | 环境架构需独立第三方审查 |

## 自动触发规则

| 触发点 | 位置 | 动作 | 模式 |
|--------|------|------|------|
| 总验证方案生成后 | Step 7 完成后 | 对验证方案执行 `debate plan` | `plan` |
| 测试点分解完成后 | Step 9 完成后 | 对测试点覆盖执行 `debate plan` | `plan` |

## 用户触发

用户可随时手动指定对抗评审：

```
"帮我用 debate 审查验证方案"         → debate plan
"用 debate review 模式检查测试点"    → debate review
"用 debate 让外部模型评审环境方案"    → debate plan
```

## 输出整合

对抗性评审的结果整合到验证方案文档中：

1. 将 debate 发现的**覆盖盲点**补充到测试点清单
2. 将**策略分歧**标记为"需人工确认"
3. 将**改进建议**补充到验证方案的风险章节
4. 对抗性发现由本 Agent 综合判定是否需要修改方案

## 执行模板

```
调用 Skill("debate", args="{模式} [--provider {provider}]")

执行后：
1. 提取 VERDICT → 判断是否通过
2. 提取 issues/critical 数量 → 汇总到问题清单
3. 跨模型分歧点 → 标记为"需人工确认"
4. debate 的 REVISE 结果 → 要求方案修订
```

**注意**：
- debate 需要外部 LLM CLI（codex/gemini/kimi/glm/mimo/claude），如无可用 Provider 则跳过并标注
- debate 有额外 API 成本，执行前需确认用户同意

## Step 2：环境配置抽象（组 A）

> 将 DUT 的所有可配置项抽象为结构化的环境配置参数表。

**抽象维度**：

| 维度 | 说明 | 示例 |
|------|------|------|
| **模块参数** | DUT 的 parameter/localparam | DATA_WIDTH=32, CH_NUM=4 |
| **工作模式** | DUT 支持的工作模式 | 正常模式/旁路模式/低功耗模式 |
| **接口配置** | 各接口的协议参数 | AXI4 32bit, APB 32bit |
| **寄存器配置** | 可编程寄存器的配置空间 | CTRL[0]=en, INT_MASK[3:0] |
| **运行时参数** | 动态可调的运行时参数 | 优先级、阈值、超时值 |
| **时钟复位** | 时钟频率、复位方式 | clk=200MHz, 异步复位 |

**输出格式**：

```markdown
## 环境配置参数表

### DUT 配置（编译时）
| 配置项 | 参数名 | 默认值 | 可选值 | 位宽 | 说明 |
|--------|--------|--------|--------|------|------|
| 数据位宽 | DATA_WIDTH | 32 | 8/16/32/64 | - | 参数化 |

### DUT 配置（运行时）
| 配置项 | 寄存器 | 位域 | 默认值 | 可选值 | 说明 |
|--------|--------|------|--------|--------|------|
| 模块使能 | CTRL | [0] | 0 | 0/1 | 运行时开关 |
```

**抽象原则**：
1. 编译时参数和运行时配置严格分离
2. 每个配置项必须有默认值和可选值范围
3. 配置项之间的依赖关系必须标注
4. 环境配置表是用例规划的基础——每个配置组合代表一个验证空间点

> 从寄存器 UA 微架构文档和寄存器 RTL 代码中提取寄存器实现细节，生成 RAL 模型规划和后门访问配置方案。本步骤输出写入组件方案（`{module}_cfg_agent`、`{module}_env`、`{module}_scoreboard`）。

### 3.1 寄存器信息提取

**从寄存器 UA 提取**：

| 提取项 | UA 来源 | 用途 |
|--------|---------|------|
| 寄存器地址映射 | UA §5.6 寄存器分配表 | RAL 地址定义 |

| 复位值 | UA §5.6 或 FS §7.2 | RAL reset value |

| 寄存器归属（本模块/功能模块） | UA §5.6 | 确定后门访问层级 |

**从寄存器 RTL 提取**：

| 提取项 | RTL 来源 | 用途 |
|--------|----------|------|
| 地址解码 case/if-else | `always @(*)` 地址解码块 | 后门 hierarchical path |

| 写保护/使能条件 | `if (wr_en && !lock)` 等条件 | 后门写约束 |

### 3.2 RAL 模型规划

**RAL 模型结构**（写入 `{module}_cfg_agent` 组件方案 §3.5）：

```systemverilog
// RAL 寄存器定义示例
class ral_reg_{module}_CTRL extends uvm_reg;
    rand uvm_reg_field module_en;
    rand uvm_reg_field int_mask;

    function new(string name = "{module}_CTRL");
        super.new(name, 32, UVM_NO_COVERAGE);
    endfunction

    virtual function void build();
        module_en = uvm_reg_field::type_id::create("module_en");
        module_en.configure(this, 1, 0, "RW", 0, 1'h0, 1, 1, 1);
        int_mask = uvm_reg_field::type_id::create("int_mask");
        int_mask.configure(this, 4, 4, "RW", 0, 4'h0, 1, 1, 1);
    endfunction
endclass

// RAL Block 定义
class ral_block_{module}_reg_block extends uvm_reg_block;
    rand ral_reg_{module}_CTRL   CTRL;
    rand ral_reg_{module}_STATUS STATUS;

    function new(string name = "{module}_reg_block");
        super.new(name, UVM_NO_COVERAGE);
    endfunction

    virtual function void build();
        default_map = create_map("default_map", 0, 4, UVM_LITTLE_ENDIAN);

        CTRL = ral_reg_{module}_CTRL::type_id::create("CTRL");
        CTRL.configure(this, null, "CTRL");
        CTRL.build();
        default_map.add_reg(CTRL, `UVM_REG_ADDR'h00`, "RW");
    endfunction
endclass
```

### 3.3 后门路径映射

**后门 hierarchical path 规则**：

| 寄存器归属 | 后门路径格式 | 说明 |
|-----------|-------------|------|
| 顶层寄存器模块 | `{tb_top}.{dut_inst}.{reg_mod}.{reg_signal}` | 直接访问寄存器存储 |
| 功能模块内寄存器 | `{tb_top}.{dut_inst}.{func_mod}.{reg_signal}` | 通过功能模块路径 |
| FIFO 阈值配置 | `{tb_top}.{dut_inst}.{func_mod}.{fifo_inst}.{threshold}` | 深层路径 |

**后门路径提取方法**：

1. 从 RTL 中 Grep `always @(posedge clk` 找到寄存器更新块
2. 提取每个寄存器的存储信号名（如 `ctrl_r`、`int_mask_r`）
3. 确定信号所在模块层级（顶层 / 子模块）
4. 拼接完整的 hierarchical path
5. 对于 W1C 寄存器，标注后门写入的特殊处理（写 1 清零 vs 直接赋值）

**后门配置输出格式**（写入 `{module}_cfg_agent` 组件方案 §3.6）：

```markdown
### 后门路径映射表

| 寄存器名 | 偏移地址 | 存储信号 | 后门路径 | 访问类型 | 特殊处理 |
|----------|----------|----------|----------|----------|----------|
| CTRL | 0x00 | ctrl_r | tb_top.dut.u_reg_mod.ctrl_r | RW | - |
| STATUS | 0x04 | status_r | tb_top.dut.u_reg_mod.status_r | RO | 只读后门 |
| INT_STATUS | 0x08 | int_status_r | tb_top.dut.u_reg_mod.int_status_r | W1C | 后门写 1 清零 |
| FIFO_THR | 0x10 | fifo_threshold | tb_top.dut.u_data_path.fifo_threshold | RW | 深层路径 |
```

### 3.4 后门访问策略

| 场景 | 访问方式 | 说明 |
|------|----------|------|
| 测试初始化（快速配置） | 后门写 | 绕过 APB 总线，直接写寄存器存储信号 |
| 状态查询（不触发副作用） | 后门读 | 读寄存器存储信号，不影响硬件状态 |
| W1C 清零 | 后门写 + 特殊逻辑 | 写 1 到对应位触发清零 |
| 复位值验证 | 后门读 + compare | 复位后立即读取，比对 FS 定义的复位值 |
| 功能测试（正常路径） | 前门（APB） | 通过总线正常读写 |
| 覆盖率采样 | 后门读 | 采样寄存器值变化，不影响功能 |

## Step 4：功能测试点分解（组 B）

> 从 FS 功能描述出发，按功能层次分解测试点。

**分解方法论**：

```
功能需求 (REQ)
  ├── 正常功能测试点
  │     ├── 单次操作
  │     ├── 连续操作
  │     └── 并发操作
  ├── 边界条件测试点
  │     ├── 最大值/最小值
  │     ├── 空/满/零
  │     └── 首/末拍
  ├── 配置变体测试点
  │     ├── 参数组合
  │     └── 模式切换
  └── 时序约束测试点
        ├── 最高频率
        ├── 最低速率
        └── 背压场景
```

> 从 FS 接口定义和 UA 接口时序出发，分解接口级测试点。

**分解维度**：

| 维度 | 说明 | 示例 |
|------|------|------|
| **协议合规** | 协议规范要求的行为 | AXI burst 类型、APB 等待周期 |

## Step 6：异常/边界测试点分解（组 B）

> 分解异常场景和边界条件测试点，这是验证最容易遗漏的部分。

**异常场景分类**：

| 类别 | 说明 | 示例 |
|------|------|------|
| **复位** | 复位期间和复位释放后的初始状态 | 复位中发起请求、复位后立即工作 |
| **超时** | 握手超时、响应超时 | valid 拉高后 ready 始终不响应 |
| **溢出** | FIFO 满时写入、计数器溢出 | 写满 FIFO 继续写入 |
| **欠载** | FIFO 空时读取、Credit 归零 | 空 FIFO 发起读 |
| **配置错误** | 非法配置组合 | 同时使能互斥模式 |
| **时钟异常** | 时钟门控、频率切换 | ICG 关闭时发起请求 |
| **中断风暴** | 连续中断触发 | 所有中断源同时触发 |
| **多源竞争** | 多通道/多源同时操作 | 所有通道同时请求仲裁 |

**边界条件**：

| 类别 | 说明 | 示例 |
|------|------|------|
| **数值边界** | 最大值、最小值、零值 | 数据=MAX、地址=0、长度=1 |
| **容量边界** | FIFO 满/空、计数器满 | FIFO 深度-1、计数器 MAX-1 |
| **时序边界** | 最高/最低速率 | 最高频率连续操作、最低速率单次操作 |
| **状态边界** | FSM 状态转移边界 | 从每个状态都能回到 IDLE |

**测试点编号规则**：`TP-EXC-{类别}-{序号}` / `TP-BND-{类别}-{序号}`

# 测试点分解方法论（核心能力）

## 分解四原则

1. **自顶向下**：功能 → 场景 → 测试点，逐层展开
2. **正交分解**：每个维度独立分解，组合时避免重复
3. **穷举覆盖**：每个功能点至少有一个正常+一个异常测试点
4. **可追溯**：每个测试点必须关联 REQ 或接口编号

## 分解模板

对于每个功能点，按以下 6 个维度展开：

| 维度 | 问题 | 输出 |
|------|------|------|
| 正常路径 | 这个功能正常怎么用？ | 基本功能测试点 |
| 边界条件 | 什么情况下会到极限？ | 边界测试点 |
| 异常输入 | 什么情况下会出错？ | 异常测试点 |
| 并发场景 | 和谁一起用会冲突？ | 并发测试点 |
| 时序约束 | 时间上有什么要求？ | 时序测试点 |
| 配置变体 | 不同配置下行为一样吗？ | 配置测试点 |

## 环境配置抽象方法

**抽象三步法**：
1. **列举**：从 FS §7（寄存器）+ UA §9（参数）提取所有可配置项
2. **分类**：按编译时/运行时、静态/动态分类
3. **约束**：标注配置项之间的依赖和互斥关系

**约束类型**：

| 约束类型 | 说明 | 示例 |
|----------|------|------|
| 值域约束 | 配置项取值范围 | DATA_WIDTH ∈ {8,16,32,64} |
| 依赖约束 | A 依赖 B | burst_len ≤ FIFO_DEPTH |
| 互斥约束 | A 和 B 不能同时 | 模式 A 使能时模式 B 必须关闭 |
| 默认约束 | 常用配置组合 | test_type=basic 时用默认配置 |
